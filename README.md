# Shell++：让手环不只是被查看，而是真正被使用

## Shell++ 的缘由

一切开始于一个看似简单的需求：**怎样更轻松地拿到手环截图？**

现有的截图方案往往需要反复连接设备、拉取日志，再从大量内容中寻找并导出截图。偶尔使用或许还能接受，但当截图变成调试、记录和分享中的高频操作时，繁琐的步骤很快就会消耗耐心。一次截图不应该伴随着一整套复杂流程，更不应该要求每位用户都先成为熟悉日志与命令的开发者。

于是，我们换了一个方向思考问题。既然截图产生在手环上，为什么不能让手环自己完成截图，再把结果直接送到用户手中？

这便是 Shell++ 最初的答案：**由手环端完成截图与本地整理，再通过 Interconnect 建立设备间连接，将截图传输到接收端。** 用户不必频繁拉取日志，也不必在层层目录中寻找文件。截图从一个需要折腾的技术操作，变成一件清晰、直观而且可以重复完成的小事。

Shell++ 也因此诞生。它不是为了给原有流程再加一层包装，而是希望重新设计手环截图从产生、管理到离开设备的完整体验。

## 手环端的工作逻辑

用户发起截图后，手环端会调用 Lua 后端捕获画面并保存到本地，同时更新截图状态与历史记录，方便用户查看和选择。

开始同步时，Shell++ 会确认连接并完成握手，再将选中的截图有序分片发送。接收端汇总进度并保存文件，批量任务则会依次处理至完成。

**发起截图，手环捕获，本地整理，选择内容，连接传输，接收保存。** 除了 Interconnect，支持的设备还可以通过局域网使用 HTTP 直传。

## 从截图工具到手环工具箱

截图是 Shell++ 的起点，却不是终点。围绕对设备的理解与探索，我们不断加入相关能力，让它逐渐成长为一套实用的手环工具箱。

### 应用管理

直观呈现设备中的应用信息与状态，让过去依赖调试命令的管理操作拥有清晰界面。

### 文件查看

浏览关键目录、截图、日志与配置，并支持文本、十六进制查看和必要的文本编辑，让排查问题更加直接。

### 设备性能测试

通过 Lua 算力测试与 MCU 浮点性能测试，用实际结果帮助用户了解设备表现。

### CPU 与内存监控

在应用内查看 CPU 与内存占用，或开启悬浮显示持续观察，为设备负载分析和功能调试提供即时反馈。

## 全平台都能使用

无论使用 Android、iOS 还是 PC，Shell++ 都希望为数据同步提供合适的路径。

Android 用户可以通过专用同步器接收、预览截图并保存到相册。非 Android 用户则可以借助 AstroBox 插件或局域网传输，完成截图拉取、选择与保存。

**平台可以不同，体验不该因此缺席。** 多种传输方式让用户无需更换主力设备，也能选择适合自己的连接方案。

## 离线也能完成重要的事

Shell++ 的许多核心能力围绕设备本地运行。截图、历史记录、文件查看、性能测试与状态监控不需要依赖持续在线的云端服务。没有网络时，用户依然可以在手环上完成操作、保存结果，并在连接条件恢复后再进行传输。

这种本地优先的设计让工具更加可靠，也让数据边界更加清晰。截图和设备文件首先属于用户自己，只有在用户主动选择同步时，内容才会沿着指定链路发送到接收端。

## 好用、好看、强大，同时向前

如果要用三个词概括我们希望 Shell++ 成为怎样的应用，那就是：**好用、好看、强大。**

好看，让用户愿意打开并继续探索；好用，让复杂能力拥有清晰、顺手的操作流程；强大，则让截图、文件、应用管理与性能监控真正解决实际问题。

**好看让人靠近，好用让人留下，强大让它值得长期使用。** 这不是三个先后完成的阶段，而是 Shell++ 始终并行推进的方向。

## 名为 Shell，更是 Plus Plus

Shell 是我们理解设备的起点，两个加号则代表持续生长。

从简化一次截图开始，到建立完整的手环工作链路，再到应用管理、文件查看、性能测试、Lua 探索与悬浮监控，Shell++ 的每一步都来自实际使用中的问题。它不是一开始就被规划成庞大的工具集合，而是在一次次需求、实验和改进中，逐渐找到自己的形态。

未来，Shell++ 仍会沿着同一个方向继续前进：让复杂能力更容易抵达，让设备数据更自由地流动，让手环从一个只能被动展示信息的终端，变成一个可以被理解、被探索、被充分使用的设备。

**Shell++，从一张截图出发，为手环打开更多可能。**
